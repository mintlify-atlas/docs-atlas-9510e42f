The FreeRTOS kernel is separated into a hardware-independent core and a set of platform-specific **ports**. A port provides the context-switch mechanism, interrupt handling, and type definitions that the kernel needs to run on a particular CPU and toolchain combination.

<Warning>
If your SDK or BSP already ships a FreeRTOS port (common with Espressif ESP-IDF, Raspberry Pi Pico SDK, and vendor Cortex-M SDKs), use that port rather than pulling one directly from this repository. Vendor ports are tuned to their specific clock and interrupt controller configurations.
</Warning>

## Directory structure

All ports live under `portable/`. The first level of subdirectories identifies the **compiler toolchain**; the second level identifies the **target architecture**.

```
portable/
├── GCC/           # GNU/Clang toolchain ports (largest collection)
├── IAR/           # IAR Embedded Workbench ports
├── RVDS/          # ARM RealView Development Suite
├── Keil/          # Keil MDK (ARMCC)
├── CCS/           # TI Code Composer Studio
├── MSVC-MingW/    # Windows simulation (x86, MSVC or MinGW)
├── ARMClang/      # Arm Compiler 6 (clang-based)
├── Renesas/       # Renesas CC-RX / e2 studio
├── MPLAB/         # Microchip MPLAB XC (PIC)
├── MemMang/       # Heap implementations heap_1.c … heap_5.c
├── Common/        # Shared utility code used by some ports
└── template/      # Skeleton port for new platform bring-up
```

Each architecture subdirectory contains exactly two files:

| File | Purpose |
|------|---------|
| `portmacro.h` | Type definitions (`StackType_t`, `BaseType_t`), critical-section macros (`portENTER_CRITICAL`, `portEXIT_CRITICAL`), yield macro (`portYIELD`), byte alignment, stack growth direction |
| `port.c` | Context-switch implementation, tick interrupt handler, and any required assembly stubs |

Some ports add a `portasm.s` or `portASM.S` assembly file alongside `port.c` when the context switch cannot be expressed in C.

## Selecting a port

### With CMake (recommended)

Set the `FREERTOS_PORT` CMake cache variable to the port string that matches your compiler and target. The variable must be set **before** `FetchContent_MakeAvailable` or `add_subdirectory` is called.

```cmake
set(FREERTOS_PORT "GCC_ARM_CM4F" CACHE STRING "" FORCE)
```

The port string combines the toolchain prefix and architecture name:

```
<COMPILER>_<ARCHITECTURE>
```

For example: `GCC_ARM_CM4F`, `IAR_ARM_CM33_NONSECURE`, `MSVC_MINGW`.

### Without CMake

Add the two port files directly to your build system:

```
portable/<Compiler>/<Architecture>/portmacro.h   (include directory)
portable/<Compiler>/<Architecture>/port.c         (source file)
portable/MemMang/heap_4.c                         (or another heap)
```

Ensure the directory containing `portmacro.h` is on your compiler's include path so that `portable.h` can find it.

| `GCC_ARM_CM3` | `GCC/ARM_CM3` | Cortex-M3 (e.g. STM32F1) |
| `GCC_ARM_CM3_MPU` | `GCC/ARM_CM3_MPU` | Cortex-M3 with MPU |
| `GCC_ARM_CM4F` | `GCC/ARM_CM4F` | Cortex-M4 with FPU (e.g. STM32F4, STM32F7-M4) |

| `GCC_ARM_CM7` | `GCC/ARM_CM7` | Cortex-M7 (e.g. STM32H7, iMXRT) |
| `GCC_ARM_CM23_NTZ_NONSECURE` | `GCC/ARM_CM23_NTZ` | Cortex-M23 (no TrustZone) |
| `GCC_ARM_CM33_NONSECURE` | `GCC/ARM_CM33` | Cortex-M33 non-secure world (e.g. STM32L5) |

| `GCC_RP2040` | ThirdParty | Raspberry Pi RP2040 (Cortex-M0+) |

## Memory management (MemMang)

The `portable/MemMang/` directory contains five heap implementations. Choose one and add it to your build alongside the port files.

| File | Strategy | Notes |
|------|----------|-------|
| `heap_1.c` | Simple bump allocator, no free | Suitable when tasks are never deleted |
| `heap_2.c` | Best-fit, allows free, no coalescence | Fragments over time; prefer heap_4 |
| `heap_3.c` | Wraps standard `malloc`/`free` | Thread-safety provided by suspending scheduler |
| `heap_4.c` | Best-fit with coalescence | Recommended general-purpose choice |
| `heap_5.c` | Like heap_4 but spans multiple memory regions | Use when RAM is non-contiguous |

A custom heap can be provided by pointing `FREERTOS_HEAP` at a `.c` file path instead of a number (see [CMake integration](/porting/cmake-integration)).

## Creating a new port

Start from `portable/template/`. It contains a minimal `portmacro.h` with all required type and macro definitions, and a stub `port.c`. The template compiles cleanly and is the port used by the `cmake_example` project in `examples/cmake_example/`.

The minimum a port must implement:

- **`portmacro.h`** — `StackType_t`, `BaseType_t`, `UBaseType_t`, `portSTACK_GROWTH`, `portBYTE_ALIGNMENT`, `portENTER_CRITICAL`, `portEXIT_CRITICAL`, `portDISABLE_INTERRUPTS`, `portENABLE_INTERRUPTS`, `portYIELD`, `portTASK_FUNCTION`
- **`port.c`** — `pxPortInitialiseStack()`, `xPortStartScheduler()`, `vPortEndScheduler()`, tick interrupt handler
- **`portasm.s`** (if needed) — context save/restore, `vPortYield` trampoline
